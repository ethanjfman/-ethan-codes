The Beginning

How it started

Ethan is 10 years old and loves Roblox. One day he asked: "Can I make my own game?" — and we said yes.

🎮

Already a player

Ethan had been playing Roblox for years — he knew what made games fun. That turned out to be a huge advantage when it came to designing his own.

🧠

Zero coding experience

He'd never written a line of code before. No Scratch, no Python, nothing. Roblox Studio and Lua were his first programming environment.

🌈

His vision

A pastel rainbow obby (obstacle course) with moving platforms, kill bricks, spinning traps — and eventually, gamepasses to earn real Robux.

💡

The big insight

Kids learn fastest when the output is something they already care about. Ethan didn't want to learn to code — he wanted to build a game. The coding came along for the ride.

The journey so far

Each session, something new clicks. Here's how it's unfolded.

Session 1 · June 2026
First scripts. First bugs. First breakthroughs.
Ethan's first day of real scripting. He got kill bricks working, then spent a full session debugging why his moving platform wouldn't move — trying BodyPosition, TweenService, and direct Position assignment before finally landing on CFrame. The platform not only moved, it carried him with it.

The biggest moment: he spotted the bug himself. His script said Part1 but his Explorer said part1. One capital letter. He found it.
Kill Bricks Moving Platforms CFrame Debugging
Session 2 · June 2026
Going public — and a bug that wasn't in the code
Big milestone: Ethan's game went Public on Roblox. We also bumped into Roblox's new publishing-fee system and worked through what it takes to reach players of all ages.

Then came the real lesson. His bouncy jump pad refused to work — no bounce, no error, nothing. We checked the code: it was perfect. So we went hunting everywhere else instead — Script vs LocalScript, the Disabled flag, CanTouch, and finally the culprit: the script was sitting in the wrong part. The takeaway that stuck: sometimes the bug isn't in your code at all.

A new superpower: making things from code
Ethan unlocked a new way of thinking: instead of only changing parts that already exist, he learned to create brand-new things from code with Instance.new() — like a "Stage" counter that pops up next to every player.

Then he started his very first money-maker: a Speed Boost gamepass. It's teaching him MarketplaceService — how a game checks whether a player has bought something and gives them a reward.

MovingPlatform.lua — Ethan's first working script
-- Ethan's moving platform
-- Written in Session 1 after trying 4 different approaches

local main = script.Parent
main.Anchored = true

local startPos = main.CFrame
local moveDistance = 50

while true do
    for i = 0, moveDistance, 0.1 do
        main.CFrame = startPos * CFrame.new(i, 0, 0)
        task.wait(0.01)
    end
    for i = moveDistance, 0, -0.1 do
        main.CFrame = startPos * CFrame.new(i, 0, 0)
        task.wait(0.01)
    end
end

For Parents

What we've learned along the way

You don't need to know how to code to support a kid who's learning. Here's what's actually worked.

1
Start with what they already love Ethan didn't want to "learn to code" — he wanted to build a Roblox game. The coding was just the tool to get there. Motivation comes from the output, not the curriculum.
2
Let them struggle (a little) The moving platform took hours across multiple approaches. That frustration — and the eventual breakthrough — was worth more than any tutorial. Don't solve it for them too quickly.
3
Real goals beat fake exercises Ethan isn't learning Lua — he's trying to earn Robux and have his friends play his game. Those stakes make every bug fix matter in a way that practice problems never could.
4
Screenshots are worth a thousand words Instead of asking Ethan to describe the problem, we ask him to take a screenshot. What's visible in the Explorer, the Properties panel, or the error output tells us everything we need to know.
5
Explain the why, not just the what When we explained why CFrame carries the player but TweenService doesn't, Ethan actually understood it. Kids this age can handle more conceptual depth than we expect.
